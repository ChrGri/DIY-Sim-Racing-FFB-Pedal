Kontext:
  Der SimHub-Plugin sendet DapAttrPacket_t mit classId_u8 = 0x02
  (DAP_ATTR_CLASS_SERVO_MODBUS) um iSV57-Register direkt aus der UI
  heraus zu lesen/schreiben. Die Firmware kennt diese Klasse noch nicht.
  Ziel: Modbus-Zugriff soll thread-safe ueber die bestehende State Machine
  in StepperWithLimits::processPendingCommands() abgewickelt werden,
  analog zum Pattern in Isv57Communication::readServoStates().
  Mehrere aufeinanderfolgende Register sollen in einem einzigen Modbus-Frame
  gelesen werden (FC03 mit count > 1), wie es readServoStates() macht.

    int Isv57Communication::readRegisters(
            uint16_t startAddr_u16, uint8_t count_u8, int16_t* out_pi16)
    {
        if (count_u8 == 0 || out_pi16 == nullptr) return -1;

        int bytesReceived = modbus.sendRequestAndReceiveResponse(
            slaveId, 0x03, (int32_t)startAddr_u16, (int32_t)count_u8);

        if (bytesReceived != (count_u8 * 2)) return -1;

        modbus.getRawRxBuffer(raw, len);
        for (uint8_t i = 0; i < count_u8; i++) {
            out_pi16[i] = modbus.convertRxBufferToInt16(i);
        }
        return (int)count_u8;
    }

DESIGN-NOTE: Das Pattern ist identisch zu readServoStates().
             sendRequestAndReceiveResponse sendet FC03 mit startAddr und
             count, empfaengt count*2 Bytes Nutzdaten.
             convertRxBufferToInt16(i) liefert Register i als int16.

    void StepperWithLimits::scheduleServoModbusCmd(
            const ServoModbusCmd_t& cmd)
    {
        // Write struct first, then set flag (acts as release fence on volatile).
        servoModbusPendingCmd_st = cmd;
        servoModbusCmdPending_b  = true;
    }

AENDERUNG B: tryGetServoModbusReadResult implementieren.

    bool StepperWithLimits::tryGetServoModbusReadResult(
            uint16_t& addrOut, int16_t* valuesOut, uint8_t& countOut)
    {
        if (!servoModbusReadReady_b) return false;
        addrOut  = servoModbusReadAddr_u16;
        countOut = servoModbusReadCount_u8;
        memcpy(valuesOut, servoModbusReadResult, countOut * sizeof(int16_t));
        servoModbusReadReady_b = false;  // consume the result
        return true;
    }

AENDERUNG C: processPendingCommands() am Ende um den neuen Block ergaenzen.
             Einfuegen direkt vor der schliessenden Klammer der Funktion.
             (Aktuell endet die Funktion nach dem updateServoParams_b-Block.)

    // --- Pending Servo Modbus Command (UI-triggered register access) ---
    // Consumed here on the servo task to avoid concurrent Serial2 access.
    if (servoModbusCmdPending_b) {
        servoModbusCmdPending_b = false;  // clear flag first

        const ServoModbusCmd_t& cmd = servoModbusPendingCmd_st;
        uint8_t count = (cmd.count_u8 > 8u) ? 8u : cmd.count_u8;

        if (cmd.isWrite_b) {
            // WRITE: Modbus FC06 schreibt immer genau 1 Register pro Frame.
            // Mehrere Register werden als aufeinanderfolgende Einzelschreibvorgaenge
            // abgearbeitet (kein FC16 Multi-Write noetig fuer kleine Anzahlen).
            for (uint8_t i = 0; i < count; i++) {
                isv57.writeHoldingRegisterToDevice(
                    isv57.slaveId,
                    (int32_t)(cmd.startAddr_u16 + i),
                    (uint16_t)cmd.values[i]);
            }

        } else {
            // READ: count aufeinanderfolgende Register in einem einzigen
            // FC03-Frame lesen (analog zu readServoStates()).
            int16_t buf[8] = {};
            int n = isv57.readRegisters(cmd.startAddr_u16, count, buf);
            if (n > 0) {
                memcpy(servoModbusReadResult, buf, n * sizeof(int16_t));
                servoModbusReadCount_u8 = (uint8_t)n;
                servoModbusReadAddr_u16 = cmd.startAddr_u16;
                servoModbusReadReady_b  = true;
            }
        }
    }

THREAD-SAFETY ANALYSE
------------------------------------------------------------------------------
  Beide Tasks (serialCommunicationTaskRx und servoCommunicationTask) laufen
  typischerweise auf dem gleichen Core (Core 0).

  servoModbusCmdPending_b ist volatile bool:
  - scheduleServoModbusCmd: schreibt struct zuerst, setzt Flag danach
    (release-Semantik durch volatile auf ESP32-S3).
  - processPendingCommands: liest Flag zuerst, liest struct danach
    (acquire-Semantik durch volatile auf ESP32-S3).
  - Auf ESP32-S3 sind bool-Writes und 32-bit-aligned Writes atomar.
  - Ein Mutex ist fuer den Prototyp nicht noetig. Fuer Produktion:
    portENTER_CRITICAL / portEXIT_CRITICAL um scheduleServoModbusCmd.

  servoModbusReadReady_b ist volatile bool (analoges handshake-Muster).

  WICHTIG: Kein direkter Modbus-Zugriff aus serialCommunicationTaskRx!
  Alle iSV57-Zugriffe bleiben im servoCommunicationTask-Context, der
  bereits alle anderen isv57.*-Aufrufe besitzt. Keine Kollision moeglich.
